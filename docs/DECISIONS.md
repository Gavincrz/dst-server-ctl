# 决策

## DST-001 | 2026-04-23 | 核心路径不使用 Docker

决定：项目核心部署路径不使用 Docker。控制器在自己的 managed root 下安装并管理原生 DST 专用服务器。

原因：目标是直接管理原生 DST 进程与受管文件布局，避免在首版引入容器层复杂度和额外部署心智负担。

## DST-002 | 2026-04-23 | Go + Svelte

决定：后端使用 Go，前端使用 Svelte。UI 构建管线接好后，发布产物应是一个嵌入静态资源的 Go 单二进制。

原因：Go 适合单二进制分发、进程控制和本地服务；Svelte 适合快速构建本地控制面板，整体仍保持轻量。

## DST-003 | 2026-04-23 | SQLite 状态存储

决定：控制器状态使用 SQLite。DST 原生配置仍以生成文件形式保存在 managed root 下。

原因：本地单机场景下 SQLite 足够简单稳妥，同时能与生成到磁盘的 DST 原生配置文件形成清晰分工。

## DST-004 | 2026-04-23 | 结构化配置为源

决定：UI 和 API 管理结构化控制器状态。DST 配置文件由该状态生成，不做原地编辑。

原因：把控制器状态和 DST 原生文本格式解耦，能减少 UI/handler 直接处理配置格式的复杂度，并保持可验证的单一真相源。

## DST-005 | 2026-04-23 | 控制器管理进程

决定：控制器直接启动、停止、监控 DST Master 和 Caves 进程。systemd 之后可以管理控制器进程本身，但首版架构中不负责 shard 生命周期。

原因：shard 生命周期是产品核心能力，不能把它外包给外部编排层，否则运行态状态和控制入口会分裂。

## DST-006 | 2026-04-24 | 控制器先启动，首次安装通过 Web 引导

决定：控制器不要求 DST 已安装完成后才可启动。首次启动时，Web UI 作为初始化入口，负责触发受管安装、展示进度、暴露错误和提供重试入口。

安装计划、状态推进和外部命令执行仍在后端完成；UI 只负责展示初始化状态，不承载安装编排逻辑。

原因：降低首次使用门槛，让用户先启动控制器再完成受管安装，同时保持安装编排职责仍在后端。

## DST-007 | 2026-05-06 | 日志流先用 SSE，不先上 WebSocket

决定：日志展示当前是单向输出场景，runtime shard、安装任务、更新任务和版本检查日志先通过 SSE 做持续推流。

原因：可以复用现有 HTTP 路由和浏览器自动重连能力，避免在还没有双向交互需求时提前引入 WebSocket 的连接管理和消息协议复杂度。

## DST-008 | 2026-05-08 | 日志流暂不升级为 file watcher

决定：日志 SSE 已经改成“首次 snapshot + 按 offset 增量读取”的服务端策略后，当前阶段暂不再把它升级为 `file watcher` 驱动。

原因：收益主要是更低延迟和更少空转，但复杂度会明显增加，需要处理 watcher 丢事件、文件轮转/替换、平台差异和额外资源生命周期。

后续如果日志连接数显著增加，或明确需要更接近实时的推送，再在 `adapter/logtail` 内部评估 watcher + 低频 fallback poll 的实现，不改变 `domain.LogStream` 和上层 HTTP/SSE 协议。

## DST-009 | 2026-05-08 | dashboard SSE 暂不承载临时操作回执

决定：dashboard SSE 目前承担“页面状态 snapshot 持续同步”职责，不承载 cluster 保存成功、运行控制成功、安装/更新按钮操作成功这类临时 UI 回执。

原因：这些回执当前不是后端持久状态，直接并入 dashboard 事件会把前端提示文案和后端协议绑死。

如果后续确实需要跨标签页或刷新后仍可见的最近操作结果，应单独定义操作事件/通知模型。继续细化 dashboard SSE 时，优先只处理真正的后端状态变化，例如 runtime history append、任务状态变化和版本状态变化。

## DST-010 | 2026-05-08 | 世界配置主源优先采用 worldgenoverride.lua

决定：服务器世界配置后续优先以 `worldgenoverride.lua` 作为人类可编辑主源，而不是先把 `leveldataoverride.lua` 当核心模型。

原因：`worldgenoverride.lua` 更适合服务端管理和后续表单映射；`leveldataoverride.lua` 更接近客户端内部保存格式，后续如需兼容导出或辅助生成，再单独评估。

这不改变“结构化状态为源、DST 文件由 adapter 生成”的总原则，只是进一步明确世界配置层的落地文件优先级。
